## 四、代码质量问题分析

### 4.1 严重问题 🔴

#### 1. App.tsx 过于庞大 (3400+ 行)

**问题**: 单个组件承担过多职责
- 画布状态管理
- 工作流状态管理
- 项目切换逻辑
- 节点操作逻辑
- 同步逻辑
- 历史记录管理
- 视频生成轮询
- ... 还有很多

**建议**: 拆分为多个自定义 Hook 和独立组件

#### 2. 类型定义混乱 (types.ts)

```typescript
// AppNode 的 data 字段包含 100+ 个可选属性
data: {
  prompt?: string;
  image?: string;
  videoUri?: string;
  // ... 100+ 个字段混在一起
}
```

**问题**: 
- 所有节点类型共用一个 `AppNode` 类型
- `data` 字段是"万能篮子"，缺乏类型安全
- 新增功能时不知道该用哪些字段

**建议**: 使用判别联合类型 (Discriminated Unions)

#### 3. 状态管理碎片化

```
- useEditorStore (Zustand) - 编辑器状态
- useUIStore (Zustand) - UI 状态
- useCanvasState (Hook) - 画布状态
- useState (Component) - 本地状态
- useRef (Component) - 引用状态
- localStorage - 持久化状态
- IndexedDB - 本地存储
- PostgreSQL - 远程存储
```

**问题**: 状态分散在多个地方，同步逻辑复杂，容易出现不一致

### 4.2 中等问题 🟡

#### 1. TODO/FIXME 遗留

发现 17 处 TODO/FIXME 注释，表明多处功能未完成：

- 视频合并逻辑
- Luma/Runway API 实现
- Sentry 错误上报
- 版本迁移
- 资产保存

#### 2. 错误处理不完善

```typescript
// 示例：很多地方只是 console.warn
} catch (error) {
  console.warn('[App] Failed to load project details', error);
  alert('加载项目失败');
}
```

用户只能看到"加载失败"，无法获取具体错误原因。

#### 3. 硬编码配置

```typescript
// 多处硬编码的超时、重试次数
const SNAP_THRESHOLD = 8;
const COLLISION_PADDING = 24;
// 5秒轮询间隔
// 10分钟超时
```

这些应该抽取为配置项。

#### 4. 注释语言混乱

代码中中英文混用，有些注释是中文，有些是英文，影响可维护性。

### 4.3 轻微问题 🟢

1. **文件命名不一致**: 有的用 PascalCase，有的用 camelCase
2. **缺少单元测试**: 只有 3 个测试文件 (`*.test.ts`)
3. **文档与代码不同步**: 部分注释与实际实现不符

## 五、架构问题诊断

### 5.1 "壳子"问题的根源

你的判断是正确的 —— 这个项目确实存在"徒有其表"的问题。原因分析：

```
┌─────────────────────────────────────────────────────────────┐
│                    V2.0 规划蓝图 (理想)                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │  工作流中心  │→│  资产中心   │→│  剧集工作区  │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
│         ↓                ↓                ↓                 │
│  ┌─────────────────────────────────────────────────┐        │
│  │              画布执行层 (底层能力)                │        │
│  └─────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    当前实现 (现实)                           │
│  ┌─────────────────────────────────────────────────┐        │
│  │              画布 + 节点 (主要实现)               │        │
│  └─────────────────────────────────────────────────┘        │
│         ↑                ↑                ↑                 │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐           │
│  │工作流中心 │    │ 资产中心  │    │ 剧集列表  │  (半成品)  │
│  │  (空壳)   │    │  (空壳)   │    │  (空壳)   │           │
│  └───────────┘    └───────────┘    └───────────┘           │
└─────────────────────────────────────────────────────────────┘
```

**核心矛盾**：
- 规划文档描述了完整的业务架构
- 实际代码主要聚焦在画布和节点层
- 上层业务逻辑只有数据结构定义，缺少实际流程串联

### 5.2 数据流断裂

```
用户操作 → UI组件 → ??? → 后端API
              ↓
         状态更新混乱
              ↓
         数据持久化不一致
```

问题表现：
1. 工作流状态存储在 `project.settings.workflowState` (JSONB)
2. 同时又有独立的 `workflow_instances` 表
3. 两者的同步逻辑不清晰

### 5.3 服务层耦合

```
App.tsx 直接调用:
├── services/api (API客户端)
├── services/syncMiddleware (同步中间件)
├── services/workflowTemplates (工作流模板)
├── services/workflow/runtime (工作流运行时)
├── services/soraService (Sora服务)
├── services/geminiService (Gemini服务)
└── ... 更多
```

**问题**: 组件层直接依赖太多服务，违反依赖倒置原则

## 九、总结

### 核心问题

1. **架构与实现脱节**: 规划文档描述了完整的业务架构，但代码主要集中在底层画布层
2. **单体组件过大**: [`App.tsx`](../App.tsx) 3400+ 行，承担过多职责
3. **类型系统混乱**: 所有节点共用一个类型，`data` 字段过于宽松
4. **状态管理分散**: 多种状态存储方式并存，同步逻辑复杂
5. **大量未实现功能**: 17+ 处 TODO/FIXME，部分 API 是空实现

### 建议行动

1. **短期**: 拆分 App.tsx，清理未实现功能，统一错误处理
2. **中期**: 重构类型系统，统一状态管理，抽象服务层
3. **长期**: 按照 V2.0 规划逐步实现工作流中心、资产管理等功能

### 风险提示

- 当前代码库体量大但质量参差，大规模重构需要谨慎
- 建议先补充测试覆盖再进行重构
- 数据迁移需要考虑向后兼容

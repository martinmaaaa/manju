# 添梯 V2.0 规划蓝图

> 状态：当前版本的主规划文档  
> 作用：作为接下来一段时间产品、架构、数据层、页面和迭代顺序的统一蓝图  
> 原则：后续涉及工作流中心、漫剧工作流、资产复用、单集画布、任务调度的改造，默认以本文件为基准推进

---

## 1. 版本目标

V2.0 的核心目标，不是继续扩展“通用无限画布”，而是把整个系统升级为一个真正的**工作流中心**。

系统主定位调整为：

- 项目默认进入工作流中心，而不是空白画布
- 漫剧工作流是一级主产品
- 画布降级为单集工作流的执行界面
- 项目级资产、剧集级执行、结果池回流形成完整闭环
- 后续新工作流以同一骨架扩展，例如漫剧解说工作流、角色资产工作流

一句话总结：

**V2.0 要把当前系统，从“节点工具集合”，升级为“围绕漫剧生产的工作流操作系统”。**

## 2. 产品目标结构

### 2.1 一级结构

- 项目中心
- 工作流中心
- 漫剧工作流
- 漫剧解说工作流
- 角色资产工作流
- 企业协作层（后置）

### 2.2 漫剧工作流主链路

- 系列设定
- 项目级资产中心
- 剧集列表
- 单集工作流
- 画布执行层
- 分镜 / 提示词 / 视频结果池
- 进度 / 回流 / 复用

### 2.3 核心产品判断

- 画布不是独立产品，而是工作流执行层
- 单集是生产执行单元
- 项目级资产是跨 80 集复用的核心资产池
- 结果池不是附件，而是镜头和资产的版本来源

## 3. 目标系统架构

### 3.1 分层架构

- 展示层：项目中心、工作流中心、资产中心、剧集列表、单集工作区
- 编排层：工作流实例、阶段状态、资产绑定、提示词编排、任务派发
- 领域层：Project / Workflow / Episode / Asset / Version / Binding / Shot / Output
- 基础设施层：PostgreSQL / 文件存储 / 对象存储 / Provider Adapter / Job Queue

### 3.2 核心关系

- 一个项目包含一个或多个工作流实例
- 一个漫剧系列工作流下包含多个单集工作流
- 项目级资产可绑定到多个单集
- 单集下包含多个镜头
- 每个镜头拥有自己的结果池
- 画布文档服务于工作流执行，而不是替代业务模型

### 3.3 设计原则

- 先业务实体，后画布快照
- 先工作流中心，后通用节点入口
- 先项目级复用，后单次生成
- 先统一任务中心，后具体平台接入
- 先创作中枢，后企业协作层

## 7. 版本边界

### 7.1 V2.0 一定要完成

- 工作流中心成为主入口
- 漫剧工作流成为主产品路径
- 项目级资产中心落地
- 剧集列表落地
- 单集工作区落地
- 画布收口到单集执行层
- 统一任务中心开始替代专用即梦任务表
- 后端开始承接工作流实体，而不是只存前端快照

### 7.2 V2.0 可兼容但不强求一次做完

- 旧 `nodes / connections` 接口兼容
- 旧 `project.settings.workflowState` 兼容
- 旧通用画布入口兼容一段时间

### 7.3 V2.0 暂缓事项

- 完整企业组织架构
- 多级审批系统
- 全量权限矩阵
- 复杂 BI 报表

## 8. 分阶段实施路线

### Phase 1：入口收口

- 项目默认进入工作流中心
- 画布不再作为主入口
- 收缩通用加号菜单

### Phase 2：数据层升级

- 增加工作流、剧集、资产、绑定、镜头、任务表
- 保留旧结构，先双写

### Phase 3：漫剧工作流定型

- 系列设定
- 项目级资产中心
- 剧集列表
- 单集工作流

### Phase 4：画布执行层升级

- 画布围绕单集执行重构
- 节点库改为阶段动作库
- 结果池接入画布右侧和底部

### Phase 5：资产复用体系

- 资产版本
- 跟随最新 / 固定版本
- 批量绑定剧集
- 连续性状态

### Phase 6：统一任务中心

- 统一即梦 / 生图 / 视频 / 配音调度
- provider 差异下沉到 adapter

### Phase 7：企业层后置升级

- 成员
- 分工
- 审核
- 统计

## 9. 当前仓库对应关系

以下模块将作为 V2.0 改造的直接基础：

- 主应用入口：`App.tsx`
- 工作流中心：`components/workflow/WorkflowCenter.tsx`
- 工作流模板：`services/workflow/registry/index.ts`
- 现有流水线模板：`services/workflowTemplates.ts`
- 项目 API：`services/api/projectApi.ts`
- 服务端数据库初始化：`server/db.js`
- 服务端持久化层：`server/persistence.js`

V2.0 的方向不是推翻这些模块，而是：

- 把它们从“前端主导 + 快照持久化”
- 升级为“业务实体主导 + 画布执行持久化”

## 10. 执行规则

从本文件创建后开始，后续版本推进默认遵守以下规则：

- 新功能优先挂在工作流中心，而不是继续扩张通用画布入口
- 漫剧相关能力优先沉淀为系列级 / 单集级工作流
- 项目级资产优先建模为可复用资产，而不是一次性节点数据
- 新生成任务优先进入统一任务中心设计
- 改造优先做兼容迁移，不做大爆炸式重构

---

## 11. 成功标准

当以下结果成立时，可认为 V2.0 目标基本达成：

- 用户从项目进入系统时，看到的是工作流中心，而不是空白画布
- 一个漫剧项目可以先建立系列资产，再批量推进剧集
- 单集工作区能够承接分镜、提示词、视频结果池
- 角色 / 场景 / 道具能够在后续剧集中稳定复用
- 即梦 / 生图 / 视频任务不再是散落逻辑，而是统一调度
- 系统结构可以继续扩展到漫剧解说工作流，而不需要重做主骨架

---

## 12. 结论

V2.0 的本质，不是“再加一些节点”，而是完成一次产品重心迁移：

**从画布工具，升级为工作流中心；从单次生成，升级为系列化生产；从零散节点，升级为项目、资产、剧集、镜头、任务组成的完整生产系统。**
